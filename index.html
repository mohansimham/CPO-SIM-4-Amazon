A quick refresher on the frameworks you'll apply in this simulation.

RICE Framework

Systematic prioritisation with four clear dimensions

RICE scores features by four factors: Reach (how many users affected), Impact (how much it moves the needle per user), Confidence (how sure you are of your estimates), and Effort (how much work it takes). The formula: (Reach × Impact × Confidence) / Effort.

How to apply it

Score each initiative on all four dimensions. Compare RICE scores to rank your backlog objectively. Use it when stakeholders disagree — it replaces opinion with structured reasoning. Be honest about Confidence — it's the most commonly inflated factor.

Example

Amazon considers adding one-click reorder to Alexa. High reach (millions of Alexa users), high impact (reduces friction), medium confidence (voice commerce is unproven), low effort (existing infrastructure). RICE says: do it.

RICE Scoring Template

  1. Reach — How many users will this impact per quarter? Use real data, not guesses.
  2. Impact — How much will it move the needle per user? 3=massive, 2=high, 1=medium, 0.5=low, 0.25=minimal.
  3. Confidence — How sure are you? 100%=hard data, 80%=good evidence, 50%=gut feel.
  4. Effort — Person-months of work. Include design, eng, QA, and launch.
  5. Score = (R x I x C) / E — Higher score = higher priority. Compare across features.
  6. Sanity Check — Does the ranking pass the smell test? If not, re-examine inputs.

MoSCoW Method

Drawing the line between essential and nice-to-have

MoSCoW categorises requirements into four buckets: Must Have (non-negotiable), Should Have (important but not critical), Could Have (nice if time allows), and Won't Have (explicitly out of scope). It forces hard conversations about what truly matters for launch.

How to apply it

Start by listing everything stakeholders want. Then force-rank: "If we shipped without this, would the product fail?" If yes, it's a Must. If it would be painful but survivable, it's a Should. Be ruthless — too many Musts means you haven't prioritised.

Example

Amazon Prime launches same-day delivery in a new city. Must Have: delivery infrastructure, driver network. Should Have: real-time tracking. Could Have: preferred delivery windows. Won't Have: drone delivery (yet).

MoSCoW Classification Template

  1. Must Have — Without this, the product doesn't work. Non-negotiable for launch.
  2. Should Have — Important but not critical. Can ship without, but plan to add soon.
  3. Could Have — Nice-to-have. Include if time/budget allows. Cut first under pressure.
  4. Won't Have (this time) — Explicitly out of scope. Prevents scope creep. Document why.
  5. Budget Rule — Must = 60% of effort. Should = 20%. Could = 20%. Won't = 0%.
  6. Stakeholder Alignment — Get sign-off on classification BEFORE development starts.

Kano Model

Understanding what delights vs. what's expected

The Kano Model classifies features into three types: Basic (expected — absence causes dissatisfaction), Performance (more is better — directly proportional to satisfaction), and Delighters (unexpected — creates disproportionate joy). Over time, delighters become basics.

How to apply it

Survey users with paired questions: "How would you feel if this feature existed?" and "How would you feel if it didn't?" The pattern of answers reveals which category the feature falls into. Invest in basics first, then performance, then delighters.

Example

For Amazon Prime: free shipping is now a Basic (expected). Fast delivery speed is Performance (faster = happier). Prime Video was originally a Delighter (unexpected bonus) — but it's becoming a Basic as competitors bundle streaming.

Kano Feature Classification

  1. Must-Be (Basic) — Expected features. Absence causes dissatisfaction. Presence doesn't delight.
  2. Performance (Linear) — More = better. Satisfaction scales proportionally with delivery.
  3. Attractive (Delighters) — Unexpected features that create disproportionate satisfaction.
  4. Indifferent — Users don't care either way. Don't waste effort here.
  5. Reverse — Features some users actively dislike. Adding them hurts satisfaction.
  6. Survey Method — Ask functional ("If feature exists?") + dysfunctional ("If it doesn't?") questions.

ICE Scoring

Fast prioritisation when you need to move quickly

ICE scores ideas on three dimensions: Impact (how much will this move the metric?), Confidence (how sure am I?), and Ease (how quickly can we ship it?). Each is scored 1-10, then multiplied. ICE is faster than RICE — ideal for rapid iteration and growth experiments.

How to apply it

Use ICE when you have a long list of ideas and need to quickly rank them. Score each 1-10 on Impact, Confidence, Ease. Multiply for a total. High-ICE items are your quick wins. Low confidence? Run a small test first before committing resources.

Example

Amazon tests adding "Frequently bought together" suggestions on product pages. Impact: 9 (drives cross-sell). Confidence: 8 (proven in retail). Ease: 7 (data exists). ICE = 504 — ship it fast.

ICE Scoring Template

  1. Impact — How much will this move the target metric? Score 1-10.
  2. Confidence — How sure are you about the impact estimate? Score 1-10.
  3. Ease — How easy is it to implement? 10=trivial, 1=massive effort.
  4. Score = (I + C + E) / 3 — Average the three. Rank all features by score.
  5. When to Use ICE vs RICE — ICE for speed (gut-level). RICE when you have real reach data.
  6. Batch & Compare — Score all candidates in one session. Relative scores matter more than absolute.
